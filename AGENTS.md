# Project Workflow Rules
- Приступать к следующему разделу плана можно СТРОГО после явного разрешения пользователя.
- **БЕЗ ЛИШНИХ ТЕЛОДВИЖЕНИЙ**: Лучше задать 3 уточняющих вопроса, чем написать гору кода, который придется переделывать.
- **ОПТИМИЗАЦИЯ И КРАТКОСТЬ**: Код должен быть чистым, сгруппированным и без дублирования. Приоритет — легкость поддержки.
- **ОТЧЕТНОСТЬ**: После каждой выполненной задачи предоставлять отчет о соответствии выполненных действий общему плану проекта.
- **ПЛАНИРОВАНИЕ И ЯЗЫК**: Всегда представлять планы на русском языке. Делать их максимально развернутыми, глубокими и подробными, не жалея символов и строк кода для обеспечения полной прозрачности и понимания контекста.
- **КАЧЕСТВО И ВЕРИФИКАЦИЯ**: Запрещено рапортовать о готовности без проверки кода (`tsc --noEmit` или `npm run lint`) и визуального подтверждения в браузере. Все импорты должны быть проверены дважды. Панель «Problems» в IDE должна оставаться АБСОЛЮТНО ПУСТОЙ (0 ошибок, 0 предупреждений).
- **ЧИСТОТА КОДА**: Строго соблюдать канонический синтаксис используемых инструментов (особенно Tailwind CSS v4). Избегать произвольных значений (arbitrary values), если существуют стандартные сокращения. Любое использование `any` в TypeScript запрещено — только строгая типизация через интерфейсы или типы.
- **БЕЗОПАСНАЯ РАЗРАБОТКА**: Перед каждым сохранением файла проверять его на наличие синтаксических ошибок, чтобы не ломать сборку.
